## 1. 背景与目标

在处理千页级大体量 PDF 文档时，传统的"全量下载 + 运行时检索高亮"模式存在严重的性能瓶颈，包括首屏加载慢、内存溢出以及高亮渲染延迟。本系统旨在解决本地环境下从前端打开 PDF、跳转指定页码，并对预定义关键词进行精准高亮的场景。

**核心目标：**
- 实现千页级 PDF 的秒级开启与跳转。
- 实现预定义关键词的精准高亮显示。
- 保证大文档滚动过程中的流畅度（虚拟渲染）。

## 2. 术语定义

- **入库高亮结果 (Ingested Highlight Result)**：PDF 在后台入库阶段提前提取的目标页码、关键词及其命中坐标数据。
- **轻量文档索引 (Lightweight Doc Index)**：存储 PDF 基础元数据（总页数、各页原始尺寸、是否线性化等）的索引，不包含全文文本。
- **线性化 PDF (Linearized PDF)**：经过 Web 优化（也称 Fast Web View）的 PDF，支持 Range 拉流，无需下载全文即可读取指定数据块。
- **虚拟渲染 (Virtual Rendering)**：前端仅对视口可见区域及其缓冲区内的页面进行 DOM 节点创建和 PDF 渲染，释放不可见区域资源。
- **Range 拉流 (HTTP Range Requests)**：利用 HTTP 协议的 Range 头，按需请求 PDF 文件的特定字节段。

## 2.1 本地试验约定

- 前端与后端都运行在本机，通过 `localhost` 或本地代理访问。
- PDF 文件与预览资源都存放在本地磁盘目录。
- 元数据与高亮结果使用 SQLite 保存。
- 不引入远程存储或远程数据库。

## 3. 范围与非目标

### 3.1 业务范围
- 仅支持入库阶段已确定的关键词及其对应页码的高亮。
- 支持千页级 PDF 的在线预览。
- 支持自动跳转至指定目标页。

### 3.2 非目标
- **不支持** 运行时全文检索（Runtime Search）。
- **不支持** 运行时 OCR 识别。
- **不支持** 前端全量文本提取。
- **不涉及** PDF 的在线编辑或批注修改。

## 4. 成功指标

- **首屏打开时间**：从请求页面到目标页（如第 800 页）首图/文本呈现，耗时 < 2 秒。
- **滚动帧率**：在快速滚动千页文档时，FPS 稳定在 50 以上，无明显白屏卡顿。
- **高亮准确率**：高亮 Overlay 与 PDF 底层文本位置完全重合，坐标偏差 < 1px。

## 5. 总体架构概览

```text
[存储层: 本地文件系统] <---- (Range 请求) ---- [前端: PDF.js + 虚拟列表]
       ^                                         |
       | (入库预处理)                             | (请求高亮数据)
       v                                         v
[后端: 处理服务] ----------------------> [数据库: SQLite 元数据/高亮表]
(线性化/坐标提取)
```

## 6. 后端设计

### 6.1 入库预处理流程
1. **解析元数据**：读取 PDF 总页数及各页尺寸（MediaBox/CropBox）。
2. **线性化处理**：使用 `qpdf` 或 `mutool` 将 PDF 转换为线性化格式。
3. **定向高亮提取**：
   - 根据预设关键词列表，扫描指定目标页。
   - 提取关键词在 PDF 内部坐标系下的矩形区域（Rects）及四元组（QuadPoints）。
4. **资源持久化**：将线性化 PDF 存入本地文件系统，索引数据存入 SQLite。

### 6.2 入库高亮结果表结构

```json
{
  "docId": "string",
  "highlights": [
    {
      "groupId": "uuid",
      "pageNumber": 800,
      "keyword": "目标词",
      "rects": [
        {"x": 100.5, "y": 200.0, "w": 50.0, "h": 12.0}
      ]
    }
  ]
}
```

### 6.3 轻量文档索引结构

```json
{
  "docId": "string",
  "totalPage": 1200,
  "isLinearized": true,
  "pageSizeList": [
    {"w": 595.28, "h": 841.89},
    {"w": 595.28, "h": 841.89}
  ]
}
```

### 6.4 线性化处理
强制要求所有入库 PDF 经过 Linearization 优化。这是实现 Range 拉流的基础，确保前端 PDF.js 能够优先获取文件尾部的 xref 表和目录信息。

### 6.5 可选：单页预览资源
针对特大型文档，可在预处理时生成目标页的低分辨率缩略图（Base64 或 WebP），随初始化接口下发，实现首屏"秒开"视觉占位。

## 7. 前端设计

### 7.1 打开流程
1. **接口请求**：获取"轻量文档索引"及"目标页高亮数据"。
2. **容器初始化**：根据 `pageSizeList` 撑起虚拟滚动条总高度。
3. **定位跳转**：根据目标页码，计算 `scrollTop` 偏移量并执行跳转。
4. **Range 加载**：PDF.js 通过 `disableAutoFetch: true` 和 `disableStream: false` 模式，仅拉取目标页相关的字节块并渲染。

### 7.2 虚拟渲染策略
- **视口管理**：只渲染 `Viewport + Buffer (上下各 2-3 页)`。
- **回收机制**：当页面滚动出 Buffer 区域时，调用 `pdfPage.cleanup()` 并销毁对应的 DOM 节点和 Canvas 实例。

### 7.3 滚动处理
- **防抖监听**：监听滚动事件，动态计算当前所在页码。
- **占位符**：未加载页面显示空白及页码 Loading 状态，维持布局稳定。

### 7.4 高亮 overlay 渲染
高亮层独立于 PDF Canvas，采用绝对定位的 div。
- **层级**：Canvas (底) < TextLayer (中) < HighlightLayer (顶)。
- **样式**：`background-color: rgba(255, 255, 0, 0.4); mix-blend-mode: multiply;`。

### 7.5 坐标转换

```javascript
const scale = viewport.scale;
const [x, y] = viewport.convertToViewportPoint(pdfX, pdfY);
```

## 8. 跨行/跨页关键词处理

- **跨行处理**：预处理阶段将一个关键词拆分为多个矩形（Rects），前端循环渲染。
- **跨页处理**：
  - 后端将关键词拆分为两个页面的高亮记录。
  - 使用相同的 `groupId` 进行关联。
  - 前端分别在两页的 Overlay 层中绘制各自的部分。

## 9. 性能策略

- **后端**：使用本地后台任务处理 PDF 线性化，避免大文件阻塞主进程。
- **前端**：启用 Web Worker 渲染 PDF 数据；使用 OffscreenCanvas 减轻主线程压力。
- **文件服务**：本地文件服务必须支持 Range 请求。
- **运行环境**：本地单机运行，前后端通过 localhost 通信。

## 10. 落地方案选型

| 方案 | 描述 | 适用场景 |
| :--- | :--- | :--- |
| **A 档 (标准版)** | PDF.js + 线性化 + SQLite 存储高亮坐标 | 通用本地试验场景 |
| **B 档 (极致版)** | 在 A 档基础上增加"单页预览图"下发 | 对首屏加载体验要求极高的场景 |
| **C 档 (轻量版)** | 不做线性化，仅做前端虚拟渲染 | 文件体积普遍较小 (< 50MB) |

## 11. 风险与缓解

- **坐标偏移风险**：部分 PDF 含有复杂的 UserUnit（缩放因子）。*缓解*：预处理时需解析 UserUnit 并标准化坐标。
- **Range 请求失败**：部分企业网关或防火墙拦截 Range 头。*缓解*：前端检测到失败后自动降级为全量下载模式。

## 12. 里程碑与排期建议

1. **第一阶段 (Week 1-2)**：本地后端预处理服务开发，实现线性化与坐标提取算法。
2. **第二阶段 (Week 3)**：SQLite 数据库设计与高亮数据接口开发。
3. **第三阶段 (Week 4-5)**：本地前端虚拟渲染引擎开发，集成 PDF.js Range 拉流模式及坐标转换模块。
4. **第四阶段 (Week 6)**：本地性能压测、边界测试（如跨页高亮）与试运行。
