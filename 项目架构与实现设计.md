# PDF 高亮定位系统：架构设计与实现设计

## 1. 目标与范围

本方案用于实现“检索列表点击后，快速打开 PDF 并定位高亮”的完整链路，采用 **per-hit** 设计：

- 列表每一项只对应一个固定高亮位置（单个矩形）。
- 前后端分离：前端 React + TypeScript + Ant Design，后端 FastAPI + Python。
- 支持千页 PDF 的可用性能（虚拟页面渲染 + Range 流式读取 + 轻量索引缓存）。

## 2. 核心设计原则

1. **每个命中一条记录（per-hit）**
   - `hit` 记录中只保存一个位置：`x, y, w, h`。
   - 跨页语义可通过 `groupId` 关联多条命中。

2. **轻量索引与列表解耦**
   - 列表接口只返回当前分页命中的业务字段。
   - `pageSizeList` 通过 `/api/pdf/{id}/meta` 单独返回，并按 `pdfId` 缓存。

3. **打开流程不阻塞**
   - 点击列表项后并行拉取 `/file` 与 `/meta`。
   - 优先渲染目标页，再进行周边页懒加载。

4. **坐标统一规范**
   - 后端存储 PDF 坐标（左下原点，点值）。
   - 前端渲染时根据 `viewport` 做坐标转换。

## 3. 总体架构

```text
前端（React）
  ├─ 列表页（分页命中）
  └─ 预览页（PDF 虚拟渲染 + 高亮）

后端（FastAPI）
  ├─ 命中查询 API（/api/highlight-hits）
  ├─ 轻量索引 API（/api/pdf/{id}/meta）
  └─ PDF 流式 API（/api/pdf/{id}/file, Range）

预处理（入库）
  ├─ PDF 线性化
  ├─ 页尺寸提取（pageSizeList）
  └─ 关键词命中提取（per-hit）

存储
  ├─ 文件系统：PDF 文件
  └─ SQLite：文档、页元数据、命中数据
```

## 4. 数据模型

### 4.1 `pdf_documents`

- `id`：文档 ID
- `file_path`：文件存储路径
- `file_name`：原始文件名
- `total_pages`：总页数
- `file_size`：文件大小
- `is_linearized`：是否线性化
- `created_at`：创建时间

### 4.2 `pdf_page_meta`

- `id`
- `pdf_id`
- `page_num`
- `width`
- `height`
- `rotation`

索引：`(pdf_id, page_num)`

### 4.3 `pdf_highlight_hits`（per-hit）

- `id`
- `pdf_id`
- `page_num`
- `keyword`
- `x, y, w, h`（单个命中矩形）
- `group_id`（可空，跨页关联）
- `created_at`

索引：`(pdf_id, page_num)`、`(pdf_id, keyword)`、`(group_id)`

## 5. API 契约

### 5.1 分页命中列表

`GET /api/highlight-hits?page=1&pageSize=20&pdfId=&keyword=`

返回示例：

```json
{
  "code": 200,
  "data": {
    "page": 1,
    "pageSize": 20,
    "total": 123,
    "items": [
      {
        "hitId": "hit_001",
        "pdfId": "doc_001",
        "fileName": "示例.pdf",
        "previewUrl": "/api/pdf/doc_001/file",
        "pageNum": 100,
        "keyword": "风险评估",
        "x": 120.5,
        "y": 210.3,
        "w": 65.0,
        "h": 12.0,
        "groupId": "grp_abc"
      }
    ]
  }
}
```

### 5.2 轻量索引

`GET /api/pdf/{id}/meta`

返回示例：

```json
{
  "code": 200,
  "data": {
    "pdfId": "doc_001",
    "totalPages": 1000,
    "fileSize": 8891223,
    "isLinearized": true,
    "pageSizeList": [
      { "pageNum": 1, "width": 595.27, "height": 841.89, "rotation": 0 }
    ]
  }
}
```

### 5.3 PDF 文件流

`GET /api/pdf/{id}/file`

- 支持 `Range` 请求，优先返回 `206 Partial Content`。

## 6. 前端实现要点

1. 列表页显示分页命中，点击后进入 Viewer。
2. 前端目录按“页面/复杂组件文件夹化”组织：`index.tsx + styles.ts + types/hooks`。
3. Viewer 并行请求：
   - PDF 文件流
   - `meta`（若缓存不存在）
4. 使用 `pdfjs-dist` 渲染页面。
5. 使用 `@tanstack/react-virtual` 做“按页虚拟渲染”。
6. 使用 `styled-components` 编写样式。
7. 高亮渲染：将 `x,y,w,h` 转换到 viewport 后绘制 overlay。
8. 点击打开时优先滚动到目标页，并预热该页前后 2 页的数据，提升大文件快速预览体验。
9. 前端统一请求层需支持：`signal`、外部取消回调、统一错误处理、流式输出。
10. PDF 文档缓存仅保留最近 5 个文档，关闭预览页时若文档仍在加载则立即取消。

## 7. 后端实现要点

1. 文档上传后执行预处理：
   - 线性化（可选 pikepdf/qpdf）
   - 页尺寸提取（PyMuPDF）
   - 关键词命中提取（per-hit）
2. 将命中、页尺寸、文档基础信息写入 SQLite。
3. 路由按职责拆分为 `pdf_meta`、`pdf_file`、`pdf_ingest` 等独立模块。
4. 列表查询做分页与关键词过滤。
5. 文件读取支持 Range，保障首开体验。

## 8. 性能策略

1. 列表不返回 `pageSizeList`，避免重复膨胀。
2. `meta` 按 `pdfId` 缓存，重复打开无需重复拉取。
3. Viewer 先渲染目标页，再按滚动加载周边页。
4. 打开时预热目标页附近的页对象，减少上下滚动等待。
5. PDF 文档缓存采用最近最少使用策略，仅保留最近 5 个文档，避免缓存无限增长。
6. 高亮坐标始终使用原始值，缩放时重算显示坐标。

## 9. 风险与规避

1. **大文档首开慢**：通过 Range + 线性化 + 虚拟渲染缓解。
2. **坐标错位**：统一坐标规范并集中做 viewport 转换。
3. **分页重复项**：按 `hitId` 去重并固定排序规则（`created_at DESC, id DESC`）。
4. **跨页语义**：通过 `groupId` 补充关联，不影响 per-hit 展示。

## 10. 交付物

1. `frontend`：前端独立项目（React + TS + styled-components）。
2. `backend`：后端独立项目（FastAPI + SQLite + PyMuPDF）。
3. 根目录本文档：统一架构与实现说明。
